## 2. 엔진 공유 전략 (별도 레포 결정에 따른 필수 설계)

크리에이터판이 별도 레포이므로 pnpm 워크스페이스 공유 불가. 다음 방식 확정:

1. **`packages/engine` + `packages/contracts`를 GitHub Packages 사설 npm으로 발행** (`@boina/engine`, `@boina/contracts`)
   - boina 레포에 발행 워크플로 추가 (버전 태그 시 자동 publish)
   - 크리에이터 레포는 npm install로 소비 — 엔진 개선이 두 판에 전파
2. 키워드 파이프라인(검증 폴더의 JS)은 **TS 패키지 `@radar/keyword-pipeline`으로 이식**하여 동일하게 발행
   - 이식 시 필수 반영: 시드 분해 폴백(검증에서 확인), 채널별 상태 리포트, 쿼터 가드
3. 발행 전 임시 경로: `pnpm add github:choiwjun/boina#path:packages/engine` 또는 로컬 `file:` 링크 — Phase 0 태스크에 포함

## 4. 착수 규칙

- **boina 레이더 모듈(04+07+08)**: boina의 거버넌스(베이스라인 patch-only)에 따라, 착수 시 07 §5 패치 항목을 boina `docs/planning/`에 Gap Review로 넣고 게이트 판정 후 진행.
- **크리에이터판(01~03, 05~06)**: 새 레포 생성 → 06-tasks Phase 0부터. 06의 태스크가 곧 백로그.
- 두 작업 모두 이 폴더 문서를 정본으로 참조하고, 변경 시 여기부터 갱신한다.

## 5. 구현 착수 전 확인해야 할 외부 전제

현재 이 폴더는 기획 정본이며, 실제 구현 소스는 별도 위치에 있어야 한다. 아래 자료가 확인되지 않으면 개발 태스크를 시작하지 않고 먼저 경로를 확보한다.

| 필요 자료 | 쓰임 | 없을 때 처리 |

| boina 모노레포 | Phase A 구현 대상. `docs/planning/` Gap Review, `packages/*`, `apps/web` 수정 | 사용자에게 boina 레포 경로 확인 |
| `사장님레이더-검증/pipeline.js` | `@radar/keyword-pipeline` TS 이식의 원본 산식 | A1-1 착수 전 원본 경로 확인 |
